# Fable Token Diet — надстройка над token-econom

token-econom действует полностью; ниже ТОЛЬКО то, что появилось/подорожало в Fable 5. При конфликте побеждает более жёсткое правило.

## 1. Workflow / мультиагентность — по умолчанию ЗАПРЕЩЕНА
- Workflow tool: только по явной команде ("ultracode", "запусти workflow", "разгони агентов"). Никогда не "задача бы выиграла от параллелизма".
- Agent tool: max 1 субагент на задачу (правило token-econom), и только когда ответ требует прочёса многих файлов. Не спавнить агента ради того, что решают 1-2 grep'а.
- Не дублировать работу делегированного агента своими поисками — жди результат.
- Fable сам склонен к оркестрации — активно дави этот рефлекс.

## 2. ToolSearch / отложенные схемы
- Схемы грузить ОДНИМ вызовом со списком: `select:A,B,C` — все инструменты, которые понадобятся на задачу. Не по одному.
- Не грузить схемы "на всякий случай". Не искать инструменты для задачи, решаемой Bash/Read.

## 3. Виджеты, артефакты, визуализация
- show_widget / Artifact — только по явной просьбе показать/нарисовать. Текст+таблица почти всегда достаточны и в 10-50x дешевле.
- read_me (visualize) не вызывать, пока не решено твёрдо, что виджет нужен.
- Скриншоты preview/computer-use: сперва preview_snapshot / preview_inspect / logs (текст). Скриншот — только как финальное визуальное доказательство по запросу.

## 4. Wake-up'ы, loop, фоновые задачи
- Фоновые задачи (Bash run_in_background, агенты) НЕ поллить — harness сам разбудит. Fallback-wakeup 1200s+.
- ScheduleWakeup: либо ≤270s (кэш жив), либо ≥1200s. Никогда 300-900s — платишь cache-miss ни за что. Каждый wake-up после 5 мин = полное перечитывание контекста.
- /loop без нужды не предлагать.

## 5. Reasoning effort
- Механическая задача (переименования, правки конфигов, однострочные фиксы) — скажи пользователю: "это уровень low effort / Sonnet, переключи" ДО начала тяжёлой работы.
- Субагентам в opts всегда задавать `effort: 'low'` для механических стадий; высокий effort — только verify/judge.

## 6. Автономный режим и длинный контекст
- После суммаризации контекста НЕ перепроверять уже установленные факты и не перечитывать файлы "для уверенности" — доверяй summary.
- Ошибка инструмента → один осмысленный ретрай с изменением, не серия слепых повторов.
- Финальное сообщение одно, в конце хода; между тул-коллами — максимум одна строка статуса, без нарратива.

## Kill-switch
"не экономь" / явный запрос на workflow, виджет, скриншот — правило снимается для этого запроса.
